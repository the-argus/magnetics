My Process

I started with the proposals, which you can find at https://people.rit.edu/ifm1110/235/git/projects/11_18_22_project3_proposal_OLD/ and https://people.rit.edu/ifm1110/235/git/projects/11_18_22_project3_proposal/

I began the project by spawning a PIXI sprite and making it move downwards all the time. Then I wrote an AABB overlap function. Then I instantiated a floor sprite and began watching the talk listed in my sources. After applying what I learned, I had a system which could handle vertical, downwards collisions (so walls and ceilings didn't work) and was extremely bouncy. I clearly had messed something up, but I decided to put it off since it was somewhat servicable. I implemented the magnets, at this point fully subscribing to an entity component system architechture. I managed to write an algorithm for getting normals, too, so collisions with walls and ceilings could now occur.

At this point I submitted to the feedback party. There was no reset button, and the incessant bouncing was a big issue. I proceeded to spend most of my time post-feedback on getting the physics right, which turned out to be easy by applying the theory I knew, instead of trying to copy and paste the equations on Catto's slides that I didn't fully understand.

Finally, I spent the last few days before submission adding the goal, fancy graphical filters, cleaning up the code, and adding more documentation. As a finishing touch, I made magnets pulse when active.

Sources

  • ES6 Class: All of the files in the "objects" folder contain classes (children of GameObject) built using an entity component system.

  • Optimized Images: My images aren't particularly optimized: they're png format (could easily be jpeg) and 1024x1024 (could probably be smaller without sacrificing much visual quality. However, there are a grand total of three (3) images, so it's not exactly negatively affecting the experience.
